The webpage for this challenge contained something new - Javascript!
This was found in the <script> element. The script had one function, which was called when the submit button was clicked.
This function was the key - it read the input from the user, and tried to compare it to the password - but very, very badly.

Instead of something simple, such as "result= pass == t" , it went the roundabout way, of creating a new string, and trying to evaluate it.
This allowed you to enter whatever you wanted, and have the script do that! Sure, you needed to engineer the input so it fits the syntax, but it works.
This is a huge weakness, a common one, and the main reason to always check user input.

This was a extreme example - indeed, the simple alternative shown above would work, and is more intuitive.
Though it would seem to be a weakness of the JavaScript, the problem shown here is the symptom of a deeper one - the user is pure evil, and only you can protect your software. It might not be as obvious as JS Injection, but whenever you deal with user input, expect the worst, and count on being surprised.

Injections aren't limited to Javascript. Whenever there is a programmer who doesn't sanitize (check) the user's input, there be injections.
In SQL, a database manipulation language, injections are commonly known, since SQL statements and queries are often built around the user's input. XKCD example.
Injections aren't alone - in C, you might be tempted to read the input from the user, into your 40-character buffer. What happens when the user input is more then 40? Terrible things.
In more recent news, the famous Heartbleed bug was caused by a similar problem - again, XKCD explains it best. In essence, the bug was caused by not checking the size of the answer requested, and internet madness occurred.
